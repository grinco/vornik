   The vornik pod has securityContext.privileged: true. If your cluster

   `pod-security.kubernetes.io/enforce: privileged` or move vornik to a
   namespace that permits it.

   The pod mounts {{ .Values.podmanHostSocket }} from its node.

   or tolerations — otherwise new pods will CrashLoopBackOff on the
   entrypoint's socket probe.

4. mTLS relay seam:
   The chart references Secret "{{ .Values.cluster.mtls.secretName }}" for the DMZ relay mTLS.
   This Secret must exist before pods start. Expected keys:
     ca.crt        — internal CA certificate
     server.crt    — worker relay-ingress server certificate
     server.key    — worker relay-ingress server private key
     client.crt    — webhook relay client certificate
     client.key    — webhook relay client private key

   Step 1 — generate the cert files (the script writes to deployments/podman/certs/):

     VORNIK_WORKER_DNS="{{ include "vornik.workerFullname" . }}" \
       bash deployments/podman/gen-cluster-certs.sh

   Step 2 — load the cert files into a Kubernetes Secret:

     kubectl create secret generic {{ .Values.cluster.mtls.secretName }} \
       -n {{ .Release.Namespace }} \
       --from-file=ca.crt=deployments/podman/certs/ca.crt \
       --from-file=server.crt=deployments/podman/certs/worker-server.crt \
       --from-file=server.key=deployments/podman/certs/worker-server.key \
       --from-file=client.crt=deployments/podman/certs/webhook-client.crt \
       --from-file=client.key=deployments/podman/certs/webhook-client.key

   Remove the local cert files after loading (never commit keys to the repo):

     rm -rf deployments/podman/certs/
